>Q1="如果有地端算力，資料洩漏高風險，企業層級 agentic ai選擇策略你有甚麼建議? "、
>Q2="我就是不想自己寫、claude、codex、openclaw這些都有desktop、或者opensource，你何以不建議接受外面寫的系統。"，
>Q3="openAI企業的agentic AI選擇是自家的codex，這對我們一般小型公司的意義是?"


整合我們在這個對話中的討論，針對「擁有地端算力、高度重視資料安全的小型企業」，我對導入 Agentic AI 的核心建議與戰略方向總結如下：

### 核心指導原則：「開源拿來主義」加上「絕對的實體隔離」

身為小型企業，你完全不需要浪費研發資源「從頭造輪子」，但也不能為了方便而輕易妥協資安。你的戰略應該是**「用開源的磚塊，在自家的斷網院子裡蓋城堡」**。

### 具體的執行與選擇策略

**1. 系統部署：拒絕分散式桌面軟體，改採「伺服器集中自託管」**

- **不建議直接安裝 Desktop App**：外部打包好的桌面端 Agent（即使是開源的）通常帶有隱藏的遙測回傳、不受控的對外連線，以及過高的本機操作權限，這會直接刺穿企業的資安防護網。
- **正確的做法（容器化與沙盒）**：採用外部寫好、成熟的開源系統（如 [Dify]()、[FastGPT]()、[OpenHands]() 等），將它們集中部署在企業內部伺服器的 Docker 容器中，並**拔除對外網際網路的連線（Air-gapped）**。限制 Agent 只能存取掛載的特定資料夾，徹底杜絕資料外洩。

**2. 模型選擇：打造企業專屬的「內部 Codex」**

- **平替策略**：既然 OpenAI 都為了保密而使用自家的 Codex，小企業更該避免將核心數據交給雲端 API。你可以利用現有的地端算力，部署強大的開源模型（如[ Qwen-Coder]()、[DeepSeek-Coder]() 或[ Llama 3.1]()），作為你們專屬的安全大腦。
- **保持彈性與防綁架**：建立在地端開源生態上，能確保你們隨時可以抽換更先進的模型或框架，避免被單一雲端供應商（Vendor Lock-in）綁架或面臨隨意漲價的風險。
- 你完全可以利用地端算力，免費用開源工具搭建出類似 OpenAI 內部的開發與代理環境。例如：
    - **程式碼輔助**：使用開源的 [Continue.dev ](#Continue.dev)或 [Tabby](#Tabby) 搭配本地大模型，讓員工在編輯器內就能享有本地端的 Copilot。
    - **自主代理**：部署開源的 [SWE-agent](#SWE-agent) 或 [OpenDevin](#OpenDevin) 於本地 Docker 沙盒，讓它們去解企業內部的 Bug 或整理報表。  
        這些外部寫好的系統，只要拔掉網路線、限制權限，就能成為小公司低成本且安全的數位員工。

**3. 資源配置：放棄「[卷](#卷)模型」，專注「[卷](#卷)工作流 (Workflow)」**

- 小公司的優勢在於靈活與專精。與其花費精力去微調模型，不如利用無代碼/低代碼的開源 Agent 框架，把公司內部的 SOP、私有資料庫與內部系統串接起來。
- 一個在本地端與企業流程深度整合的 14B 開源小模型，在解決你們的特定業務問題上，實用性與安全性絕對會大於外部通用的 GPT-4。

**4. 權限管控：實施「零信任」與「人為審批 (Human-in-the-loop)」**

- Agent 必須繼承發起員工的系統權限，絕不給予最高管理員權限。
- 在涉及資料庫修改、預算核准或任何高風險操作時，必須在系統流程中設計「中斷點」，強制要求人類主管點擊確認後，Agent 才能放行執行。

**總結來說**： 你不必自己寫系統，市面上已經有足夠好的開源工具。你的核心任務只有一個：**搭建一個斷網的沙盒環境，把這些強大的開源 AI 工具關進去，然後安全地享受它們帶來的生產力。** 這是目前小型企業在平衡「開發成本」與「極致資安」下，最務實且最具效益的解法。


## Terminology

### OpenDevin

OpenDevin 是一個開源專案，旨在模擬 Devin AI——一名能自主執行複雜工程任務的人工智慧軟體工程師。它提供透明度、可自我托管及高度可擴展性，成為企業與開發者實現合規性與客製化的理想選擇。儘管初期使用體驗如撰寫貪吃蛇遊戲時仍存技術瓶頸（如安裝迴路問題），其多代理自動規劃機制帶來極大啟發。

2025年後，社群普遍認定其為 Devin 的頂級開源替代方案。
[# OpenDevin vs Devin AI：您應該選擇哪一個「AI軟體工程師」？](https://sider.ai/zh-TW/blog/ai-tools/opendevin-vs-devin-ai-which-ai-software-engineer-should-you-bet-on)

### Continue.dev

[Continue.dev ](https://www.continue.dev/)是一款開源的 AI 程式助手，作為 VS Code 插件提供，可協助開發者撰寫程式碼。它類似於 GitHub Copilot，但強調源碼控制下的持續整合與品質驗證，能在每次拉取請求中自動執行由 AI 驅動的檢查，確保代碼符合既定標準。支援連接 Slack、Sentry、Snyk 等工具，適用於團隊協作，並提供進階模型信用額度購買功能，提升開發效率與程式碼品質。

### Tabby

Tabby 是一款免費開源的終端模擬器，支援 SSH、本地及 Telnet 連接，可替代 Windows 標準終端、PowerShell ISE、PuTTY 及 macOS Terminal.app。它不提供新殼層或 MinGW，而是聚焦於現代化、高效的終端體驗。

此外，還有一款同名的開源自托管 [AI 編碼助手](https://tabby.tabbyml.com/docs/welcome/)，可為團隊部署基於大語言模型的程式碼自動補全服務，提升開發效率。兩者雖名稱相同，但功能領域不同。


### SWE-agent

SWE-agent (software engineering, [github-star19k](https://github.com/swe-agent/swe-agent))是一個讓語言模型（如 GPT-4o 或 Claude Sonnet 4）能自主使用工具，於真實 GitHub 倉庫中自動修復問題的系統。它支援自動化軟體工程任務，例如偵測與修復程式錯誤、管理版本控制與提交變更，提升開發效率。此系統由[ J Yang 於 2024 ](https://arxiv.org/abs/2405.15793)年提出，已獲廣泛引用，展現 AI 在實際開發流程中具備實用潛力。其核心在於讓大語言模型具備操作計算機環境的能力，實現端到端的自動化程式維護。現生出[mini-swe-agent](https://github.com/SWE-agent/mini-swe-agent/)更小、更上手。


### 卷

- 「卷」指的是過度投入資源進行效益遞減的競爭（源自「內卷」）。
- 核心論點：小公司不應在「打造最聰明的模型」上內耗，而要把資源放在「設計與整合能落地的 AI 工作流」上。

#### 從「競爭大腦」轉向「競爭工作流」

OpenAI 擁有世界上最強的「AI 大腦」，小公司不可能在基礎模型上與其競爭。但是，Agentic AI 的價值不僅在於大腦，更在於**「AI 如何與你的企業流程結合」**。

- **對小公司的意義**：你不需要去追求或微調出一個極致完美的模型。你的優勢在於企業足夠小、夠靈活。你應該把精力花在**部署現成的開源框架（如前面提到的 Dify 或是 OpenHands）**，把你們公司特有的 SOP、私有資料庫與 API 串接起來。只要工作流程（Workflow）設計得好，一個 7B 或 14B 的免費開源小模型，在特定任務上的表現完全能超越沒有經過企業內部資訊定製的通用版 GPT-4。

「放棄卷模型，專注卷工作流」的核心概念是：**小公司不該在「打造最聰明的 AI 大腦」上浪費資源，而應該把精力放在「設計最完美的 AI 辦公流程」上。**

具體來說包含以下兩個層面：

#### 1. 放棄「卷」模型 (Stop competing on Models)

訓練、大幅度微調（Fine-tuning）或持續優化底層的語言模型，是科技巨頭們投入上萬張頂級顯示卡與無數科學家在玩的金錢遊戲。

如果小企業耗費大量工程師的時間與算力，試圖把一個開源模型微調到「完美」，或者企圖讓它把公司的所有知識都「背」下來，不僅成本極高，而且模型更新迭代太快，幾個月後大廠釋出新一代開源模型，你投入的努力就歸零了。現在頂級的開源模型（如 Llama 3.1、Qwen 2.5）出廠時的邏輯能力已經足夠強大，直接拿來用即可。

#### 2. 專注「卷」工作流 (Focus on Workflows)

Agentic AI 真正能在企業內發揮價值的關鍵，不在於它單純的「智商」有多高，而在於它「如何與你的**業務系統結合**」。把精力投資在構建 Workflow 上，指的是：

- **賦予工具，而不是教它死記**：與其微調模型讓它記住產品價格，不如寫一個簡單的 API（工具），讓 AI 在工作流中知道「遇到價格問題時，去查詢公司內部的資料庫」。
- **拆解 SOP，建立自動化流水線**：利用 Dify、LangGraph 等框架，把複雜任務拆成精細的步驟。例如，不要給 AI 一個空泛的指令「寫一份客戶分析報告」，而是設計一套工作流：
    1. Agent A 負責去 CRM 系統撈取客戶購買紀錄。
    2. Agent B 負責將數據進行統計分類。
    3. Agent C 負責把數據套入公司標準的報告模板。
    4. 最後交由人類主管審核。
- **防呆與容錯設計**：在工作流中設置安全閥，例如規定 AI 生成的 SQL 語法在執行前，必須經過另一個專門檢查安全性的 Agent 確認。

**總結一個比喻：** 「放棄卷模型」就像是**不要試圖自己去基因改造出一個智商 180 的超級天才員工**。 「專注卷工作流」則是**直接雇請一個智商 120 的普通優秀畢業生（現成的開源模型），然後為他準備好最齊全的工具箱、最清晰詳盡的公司 SOP 手冊、以及完善的防呆機制（Workflow）**。

對於資源有限但高度重視資安的小公司來說，後者的做法不僅成本低、見效快，而且產出的結果 100% 符合企業內部的規範。
